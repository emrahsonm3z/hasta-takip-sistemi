# Languages (TR / EN)

This document explains how the application shows everything in both Turkish
and English — the interface, the dates, the sorting, and these documentation
pages. The headline for everyone: **one click switches the whole app**; the
sections below show developers each piece of that machinery.

---

## One switch, everything follows

The TR/EN chip calls `i18n.changeLanguage`, and a single listener fans the
change out to every system that cares (`src/plugins/i18n.ts`, as shipped):

```ts
function syncLanguageCompanions(language: string): void {
  const base = language.split('-')[0]
  setPrimeReactLocale(base)   // component texts (paginator, calendar, …)
  setDayjsLocale(base)        // date formatting
  if (typeof document !== 'undefined') {
    document.documentElement.lang = base   // accessibility: <html lang>
  }
}

i18n.on('languageChanged', syncLanguageCompanions)
```

```
AppLanguageSwitcher chip
  → i18n.changeLanguage('en' | 'tr')
      → react-i18next re-renders every t() text
      → PrimeReact locale switches (its own component strings)
      → Day.js locale switches (date output)
      → <html lang> updates
      → the docs viewer re-queries content in the new language
```

The default language is Turkish; English is the fallback.

## Where the words live

Every visible sentence lives in `src/locales/tr.json` and `en.json` — always
with the same set of keys (a `node:test` fails if the two files ever drift).
Writing a sentence directly into the code is rejected by lint; the gaps lint
can't see (toasts, validation messages) are closed by **typing**: those APIs
accept only a `TranslationKey`, the union of every key path derived from
`en.json`. A typo in a key is a compile error.

```ts
t('docs.title')          // ✓ checked against the locale shape
t('docs.tittle')         // ✗ compile error
notify.success('Saved!') // ✗ compile error — not a TranslationKey
```

## Codes in the data, labels in the locales

Enum-like values are stored as stable English **codes**; their **labels** are
locale entries. The codes live in the locale files, and the patient model
(`PatientStatus`, `PatientPriority`, `PatientDepartment` unions) carries
exactly the same codes — the mapper converts the API's Turkish display
values into them at seed time:

| Code (stored) | TR label | EN label |
| --- | --- | --- |
| `waiting` | Bekliyor | Waiting |
| `inExamination` | Muayenede | In examination |
| `completed` | Tamamlandı | Completed |
| `cancelled` | İptal edildi | Cancelled |
| `urgent` / `normal` | Acil / Normal | Urgent / Normal |
| `internalMedicine`, `cardiology`, … | Dahiliye, Kardiyoloji, … | Internal medicine, Cardiology, … |

So a status renders as `t('patients.status.waiting')` — never as a hardcoded
word. Blood types keep their raw notation as keys (`0+`, `A-`, …); the EN
labels use the letter O while TR keeps the zero.

## Bilingual data fields

Some patient fields exist in both languages **in the data itself** (`note_tr`
/ `note_en`, `diagnosis_tr` / `diagnosis_en` in the API). Display picks the
active language with a Turkish fallback — one tiny pure function
(`src/lib/pickLocalized.ts`, as shipped):

```ts
export const pickLocalized = (tr: string, en: string, language: string): string => {
  const base = language.split('-')[0]
  return base === 'en' ? en || tr : tr || en
}
```

Worked example: `noteTr = "Kontrol randevusu"`, `noteEn = ""` → English UI
shows "Kontrol randevusu" (fallback), Turkish UI shows the same. Forms will
show both language fields side by side, no tabs (1.3).

## Turkish-aware text

Turkish has letters and casing rules that naive string comparison gets wrong
(`İ`/`i`, `I`/`ı`, `ğ`, `ş`). All search and sorting goes through
`src/lib/text.ts` (as shipped):

```ts
export function normalizeTurkish(value: string): string {
  return value.normalize('NFC').toLocaleLowerCase('tr')
}

const turkishCollator = new Intl.Collator('tr', { numeric: true })

export function compareTurkish(a: string, b: string): number {
  return turkishCollator.compare(a, b)
}
```

The `nfcContains` filter registered in `plugins/primereact.ts` plugs
`turkishIncludes` into every `AppDataTable` search box, so searching "ıs"
finds "Işık". The collator's `numeric: true` also sorts "2" before "10".

## Dates

Dates are ISO strings in the data and are formatted **only** through
`formatDate` (`src/lib/date.ts`, as shipped) — scattered `toLocaleString`
calls are forbidden:

```ts
export function formatDate(value: string | null | undefined, pattern = 'L'): string {
  if (!value) {
    return ''
  }
  const parsed = dayjs(value)
  return parsed.isValid() ? parsed.format(pattern) : ''
}
```

| Field | Pattern | TR output | EN output |
| --- | --- | --- | --- |
| birth date | `L` | 24.03.1987 | 03/24/1987 |
| appointment | `LLL` | 5 Haziran 2026 14:30 | June 5, 2026 2:30 PM |

Invalid or missing input renders as an empty string, never "Invalid Date".

## These docs are bilingual too

Every documentation page exists as `docs/en/<NAME>.md` and
`docs/tr/<NAME>.md`; the viewer picks the file for the active language and
swaps it live when you switch. The one exception is the Changelog —
generated in English by the release tooling and shown in English in both
languages. Prose is translated; code, file names, and identifiers stay
English in both.
